Platform Overview

The Ethiopian National Unified E-Government Platform is a centralized, enterprise-grade API ecosystem designed to enable seamless digital service delivery across government institutions. Built on a microservices architecture, the platform provides a single, standardized integration layer for orchestrating multi-agency workflows and delivering end-to-end e-government services.

The platform abstracts complex backend systemsâ€”including national identity management, ministry-specific systems, payment gateways, notifications, document management, and compliance monitoringâ€”into a unified RESTful API. This enables government entities and authorized integrators to build, deploy, and scale digital public services efficiently while ensuring security, interoperability, and regulatory compliance.

ðŸ“¢ Unified Platform: This documentation covers all e-government services exposed through a single unified API, including Authentication (VeriFayda 2.0 and Local Registration), Application Management, Notification Services, Payments, and Ministry System Integrations.

Key Features

Dual Authentication Secure identity verification using VeriFayda 2.0 (Fayda ID) for Ethiopian citizens and local registration mechanisms for foreign nationals, ensuring inclusive access to government services.
Application Lifecycle Management End-to-end management of service requests, from submission and validation to multi-level review, approval, and final service delivery, with real-time status tracking and transparency.
Ministry Integration Standardized and secure integration with ministry-specific APIs, legacy government systems, and data repositories, enabling seamless inter-agency collaboration and data exchange.
Workflow Orchestration Distributed workflow execution using the Saga pattern, ensuring transactional consistency, fault tolerance, and reliability across multiple independent services.
Multi-Channel Notifications Centralized notification delivery across email, SMS, push notifications, and in-app alerts, with template management and delivery tracking for consistent citizen communication.
Payment Integration Secure and compliant integration with government-approved payment gateways, supporting service fees, transaction reconciliation, and financial traceability.
Document Management Scalable object storage for certificates, supporting documents, and digital assets, with access control, versioning, and secure retrieval
Audit & Compliance Comprehensive audit logging and centralized log management using the ELK Stack, combined with real-time metrics collection via Prometheus and visualization and alerting through Grafana. These capabilities enable full traceability, proactive monitoring, SLA enforcement, regulatory compliance, and rapid incident response across all integrated services.

System Architecture

Microservices-based with event-driven communication

The National Unified E-Government Platform is built on a microservices-based, event-driven architecture designed for scalability, resilience, and cross-agency interoperability. Services communicate through synchronous REST APIs and asynchronous events, enabling reliable orchestration of long-running, multi-agency workflows. The architecture emphasizes security, observability, auditability, and fault tolerance, ensuring consistent service delivery across government systems.

(Government Services Catalog, Dynamic Forms, and Application APIs)

Multi-Language Support

ðŸŒ Internationalization (i18n): All user-facing text (label, help_text, title) supports multi-language dictionaries using language codes (en, am, etc.). UI clients select the appropriate language version based on user locale preferences. Form submissions include language_code to track which language variant was used.

Conditional Logic Details

ðŸ’¡ Conditional Logic: Sections and fields support conditional_on directives that show or hide content based on prior answers. Conditions can reference any earlier field using operators: equals, in, greater_than, less_than. Nested conditions (AND / OR) are supported to a maximum depth of three levels.

Service-to-Provider Mapping

Note: Service-to-provider mapping is now handled in the Service Provider Management section (Administration). Services can be provided by ministries, government agencies, NGOs, and other authorized organizations. Provider assignment and SLA management is centralized for better governance.

ðŸ“‹ Provider-Service Relationship: The service-to-provider mapping, including regional availability (zones, woredas, kebeles), is now managed in Service Provider Management section under Administration. This centralized approach ensures consistent SLA enforcement and provider lifecycle management across all services.

5.2 VeriFayda 2.0 â€” OIDC Authorization-Code Flow with PKCE

VeriFayda uses the OAuth 2.0 Authorization Code Flow with PKCE. PKCE guarantees that an intercepted authorization code cannot be exchanged for tokens without the original code verifier.

Complete Flow â€” Seven Steps

Step Action Detail
1 Generate PKCE Parameters Create code_verifier (43â€“128 chars) and code_challenge (SHA-256, Base64url)
2 Authorization Request Redirect user to VeriFayda with client_id, redirect_uri, scope, code_challenge
3 User Authentication User authenticates via OTP, biometrics, or Yes/No on VeriFayda
4 Authorization Callback VeriFayda redirects to callback URL with authorization code
5 Token Exchange Exchange code for tokens using code_verifier and client_assertion JWT
6 Retrieve User Info Use access_token to fetch user profile from UserInfo endpoint
7 Decode & Render Decode JWT to extract user claims and render in the application
ðŸ” PKCE Security Guarantee: PKCE ensures that even if the authorization code is intercepted, it cannot be exchanged for tokens without the original code_verifier. Always validate the state parameter on callback to prevent CSRF attacks.

Example â€” Generating the Signed JWT (Node.js / jose)

JavaScript
import { SignJWT, importJWK } from 'jose';
const { CLIENT_ID, TOKEN_ENDPOINT, PRIVATE_KEY_BASE64 } = process.env;

const jwk  = JSON.parse(Buffer.from(PRIVATE_KEY_BASE64, 'base64').toString('utf8'));
const key  = await importJWK(jwk, 'RS256');

const jwt  = await new SignJWT({ iss: CLIENT_ID, sub: CLIENT_ID, aud: TOKEN_ENDPOINT })
               .setProtectedHeader({ alg: 'RS256', typ: 'JWT' })
               .setIssuedAt()
               .setExpirationTime('2h')
               .sign(key);
